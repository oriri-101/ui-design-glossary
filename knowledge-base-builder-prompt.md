# 通用知识库建设 Prompt

> 适用于：让 AI 在任何专业方向上，从零构建一个高质量、可交互的知识词典 HTML 产品。
> 提炼自：ORI's UI Design Dictionary + Writing Methodology Dictionary 两个成功范式。
> 版本：v1.0 | 2026-04-15

---

## Prompt 正文

```
你是一位专业知识库架构师。你的任务是为【{领域名称}】构建一部系统化的知识词典，输出为单文件自包含 HTML（CSS/JS 不分离），可直接在浏览器中使用。

---

## 一、产品定位与核心价值

这部词典的核心目标是：
1. **个人知识资产化**——把碎片化的专业知识结构化为可检索、可引用、可持续更新的知识体系
2. **AI 校准工具**——每个条目配有 AI Prompt 用法，可直接注入 system prompt 或对话中调用
3. **认知升级载体**——不是百科全书式罗列，而是经过筛选、提炼、配实战案例的「精华版」

**产品形态**：一个有密码保护的、支持深浅色切换的、带侧边导航和全局搜索的专业知识词典。

---

## 二、信息架构（按你的专业判断填充）

请按以下结构设计知识库：

### 2.1 章节规划
- 将整个领域拆分为 **5-9 个互不重叠的章节**（遵循 MECE 原则）
- 每个章节需有：章节编号（01-09）、中文标题、一句话描述（定义这个章节解决什么问题）、SVG 图标
- 章节之间的排列遵循**认知递进**逻辑：基础 → 进阶 → 高阶 → 跨域应用

### 2.2 条目设计（每章 5-10 个条目）
每个条目是一个可展开的卡片，包含以下字段：

| 字段 | 必选 | 说明 |
|------|------|------|
| `name` | 是 | 中文术语名 |
| `en` | 是 | 英文术语名 |
| `stars` | 是 | 重要性星级（1-5），决定排序权重 |
| `desc` | 是 | 核心定义，200字以内。要求：一句话能说清本质，用 `<strong>` 标记关键概念 |
| `demo` | 是 | 实战演示区。根据条目类型选用最佳展示形式（见下方「Demo 类型库」）|
| `updated` | 是 | 最后更新日期（YYYY-MM-DD），用于追踪条目时效性 |
| `books` | 否 | 经典参考书籍，含 `title`/`author`/`url`/`quote` |
| `quotes` | 否 | 1-2 句记忆金句，要求朗朗上口、有画面感、适合在工作中引用 |

### 2.3 Demo 类型库（按条目特性选用）

根据不同知识类型，Demo 区应选用最能「秒懂」的展示形式：

- **正反对照表**（`.cmp-table`）：适用于有明确好坏之分的概念（好/差、正确/错误）
- **流程步骤条**（`.flow-steps`）：适用于有序步骤的方法论（如 AIDA、SCQA）
- **代码/公式块**（`.code-block`）：适用于可直接复用的模板、AI Prompt 指令
- **Callout 提示框**（`.callout`）：适用于核心洞察、注意事项、检验标准
- **可视化图示**（`.diagram`）：适用于需要空间理解的模型（金字塔、矩阵、循环图）
- **可交互组件**：适用于 UI/设计类条目（按钮、输入框、动效等实际渲染）
- **Tag 标签分段**（`.dtag`）：适用于多步骤拆解配文字说明的场景

**每个条目至少使用 2 种 Demo 形式**，避免全篇只有表格或只有文字。

## 三、设计规范（硬性要求）

### 3.1 视觉体系
- **设计基调**：去 AI 化、沉稳商务。不使用 emoji 作为图标，全部使用 SVG 内联图标（Feather/Lucide 风格，`fill="none" stroke="currentColor" stroke-width="2"`）
- **配色**：
  - 浅色模式：暖白背景 `#f8f7f4`，强调色赤褐 `#c45c3e`，辅助色墨绿 `#2d6a4f`
  - 深色模式：深灰背景 `#141413`，强调色提亮 `#e07a5f`，辅助色翡翠 `#52b788`
- **字体栈**：`-apple-system, BlinkMacSystemFont, "SF Pro Text", "Helvetica Neue", "Noto Sans SC", sans-serif`
- **正文**：15px / 行高 1.7 | **h1**：32px | **h2**：22px | **h3**：17px
- **圆角**：卡片 10px、按钮 6-8px、输入框 6px
- **阴影**：浅色 `0 1px 3px rgba(0,0,0,.06)` / 深色 `0 1px 3px rgba(0,0,0,.3)`

### 3.2 布局结构
```
┌─────────────────────────────────────────────────┐
│  顶栏 56px（毛玻璃效果）  [词典名称]  [深浅切换] │
├────────────┬────────────────────────────────────┤
│ 侧边导航   │                                    │
│ 240px      │  主内容区                           │
│ - 搜索框   │  max-width: 960px                  │
│ - 章节分组  │  padding: 80px 48px                │
│ - 条目链接  │                                    │
│            │  Hero 区：标题 + 描述 + Tag 统计     │
│            │  章节 → 条目卡片（手风琴展开）       │
│            │                                    │
│            │            [回到顶部]               │
├────────────┴────────────────────────────────────┤
│  响应式：768px 以下隐藏侧边栏                     │
└─────────────────────────────────────────────────┘
```

### 3.3 交互功能
1. **密码锁屏**：进入前需输入密码，支持 sessionStorage 保持登录状态
2. **深浅色切换**：右上角按钮，SVG 图标切换（月亮/太阳），localStorage 记忆偏好
3. **全局搜索**：侧边栏输入框，实时过滤条目（匹配中文名/英文名/描述）
4. **手风琴卡片**：点击展开/收起，带 fadeIn 动画
5. **侧边导航高亮**：滚动时自动高亮当前章节，点击跳转
6. **回到顶部**：滚动超过 400px 显示，带平滑滚动
7. **星级排序**：条目按 stars 降序排列，高重要性条目在前

## 五、内容质量标准

### 5.1 条目筛选准则
- **不做百科全书**：只收录「学了能直接用」的知识，淘汰纯理论/过于小众的条目
- **每个条目的检验问题**：「一个中高级从业者，知道这个概念后，下周的工作方式会改变吗？」如果不会，不收录
- **覆盖完整度**：确保行业核心概念无遗漏，但可以分批建设（v1.0 先做 40-60 个核心条目）

### 5.2 描述写作规范
- **一句话定义**：开头第一句必须能独立成立，说清这个概念「是什么」
- **来源溯源**：如有经典提出者，必须标注（如「Daniel Kahneman 提出」）
- **去 AI 味**：禁止使用「在当今快速变化的时代」「非常重要」等空泛表述
- **信息密度**：每句话都有增量信息，删掉后应影响理解

### 5.3 Demo 质量要求
- **具体性**：案例用真实行业场景（但脱敏），不用 `foo/bar/某公司`
- **对比性**：优先使用正反对照（好 vs 差），让读者 3 秒看到差异
- **可复用性**：AI Prompt 用法区域的指令可以直接复制粘贴到实际工作中

### 5.4 金句创作标准
- 朗朗上口，有节奏感（短句优先）
- 有画面感或类比（不是抽象总结）
- 能在工作会议中直接引用不尴尬

## 六、举例脱敏规则（P0 级安全要求）

所有条目中的案例和示例，严禁出现可识别的真实个人信息：
- 真实姓名 → 虚构角色名
- 公司名 + 职级组合 → 泛化描述
- 具体资产数额 → 量级描述
- 三重可交叉定位的信息组合 → 最多保留一项

替代策略优先级：角色化 > 泛化 > 变形 > 创造力优先

## 七、执行流程

### Phase 1：架构设计
1. 分析目标领域，输出章节规划（5-9 章）+ 条目清单（每章 5-10 个）
2. 定义章节之间的认知递进关系
3. 确认总条目数和 v1.0 范围

### Phase 2：内容填充
4. 按章节顺序逐一编写条目数据（DATA 数组）
5. 每个条目必须包含：定义 + Demo + 至少 1 个实战案例
6. 高星条目（4-5星）必须配备 books 和 quotes 字段

### Phase 3：产品集成
7. 将数据注入 HTML 模板，确保所有交互功能正常
8. 测试深浅色切换、搜索过滤、手风琴展开
9. 设置密码保护

### Phase 4：质量校验
10. 遍历所有条目，检查脱敏合规性
11. 验证移动端响应式布局
12. 确认所有 SVG 图标渲染正常，无 emoji 残留

---

## 八、输出要求

- 输出为**单个 HTML 文件**，CSS/JS 全部内联
- 文件大小建议控制在 200KB 以内（纯文本，不含外部资源）
- 密码默认值为 `{你指定的密码}`，使用 sessionStorage 保持状态
- Hero 区显示：标题 + 领域描述 + Tag 统计（X 大章节 / Y 个条目 / 实战案例 / AI Prompt / 深色模式）

---

## 使用说明

### 如何使用这段 Prompt

1. **替换占位符**：将 `{领域名称}` 替换为你要建设的具体领域，例如：
   - `产品经理方法论`
   - `数据分析与可视化`
   - `投资分析框架`
   - `AI Prompt 工程`
   - `项目管理知识体系`
   - `B端产品设计模式`

2. **替换密码**：将 `{你指定的密码}` 替换为你想要的访问密码

3. **可选调整**：
   - 如果领域偏实操（如编程），可在 Demo 类型库中增加「可运行代码块」
   - 如果领域偏理论（如哲学），可增加「经典原文引用区」
   - 如果领域偏视觉（如设计），可增加「可交互组件渲染区」

### Prompt 设计的核心理念

这段 Prompt 提炼了两个已验证的知识库产品的共性基因：

| 维度 | 设计决策 | 为什么 |
|------|---------|--------|
| 信息架构 | MECE 章节 + 认知递进排列 | 确保知识无遗漏且学习曲线合理 |
| 条目粒度 | 200字定义 + 多形式 Demo | 兼顾速查和深度理解两种使用场景 |
| 质量控制 | 星级排序 + 「下周能用吗」筛选标准 | 淘汰学术味重但实操性弱的条目 |
| 视觉体系 | 去 AI 化 + SVG + 暖色商务 | 避免「AI 生成感」，提升专业可信度 |
| 交互设计 | 密码锁 + 搜索 + 手风琴 + 深浅切换 | 覆盖隐私保护、快速查找、渐进阅读三大场景 |
| 持续更新 | `updated` 字段 + 分批建设 | 知识库是活的，不是一次性产出 |
| AI 友好 | 每个条目配 Prompt 用法 | 知识不只是「知道」，更是能注入 AI 工作流 |

### 迭代方向

v1.0 之后，可以考虑的扩展方向：
- **配色方案库**：如 UI 词典中的 12 套配色方案 + AI Prompt 模板
- **测验模式**：随机抽取条目做记忆测试
- **导出功能**：将条目导出为 Markdown / Anki 卡片
- **版本对比**：追踪条目的历史修改记录
- **协作标注**：多人共建时的评论和标注功能
